# Start Project

D:\AX 트레이딩 센터에 새 도메인을 추가하는 표준 진입 루틴.

## 작동 원칙

- 사용자는 D:\AX 아래 다중 프로젝트(kiwoom/wti/crypto/us/japan/quant/medical/poker/english/unity 등)를 운영 중. 각 도메인은 폴더로 격리되며 hub/이 메타 인덱스.
- 새 도메인이 외부 데이터를 사용한다면 **다른 도메인이 이미 가진 자원을 중복 구축하지 않는 것이 최우선**. KRX API가 quant에만 있다는 사실을 모르고 다른 곳에서 또 만드는 일이 발생한 적 있음(2026-05-16 사용자 지적).
- 빈 .py 파일을 미리 만들어 두지 않는다 — 구조는 인터랙티브로 결정.
- 트레이딩 도메인이 아닐 수도 있다 (의료/포커/영어/Unity 같은 비트레이딩). 인터뷰 답변으로 분류.

## 단계 0 — 사용자 트리거 해석

- `/start-project [프로젝트명]` — 명시 호출. 프로젝트명 인자 있으면 그대로 사용.
- 자연어 표현: "새 프로젝트 시작하자", "○○ 프로젝트 만들고 싶어", "○○ 폴더 새로 파줘" 등.
- 프로젝트명이 명시되지 않았으면 가장 먼저 물어본다. 한 단어 영어 소문자 권장 (디렉토리명으로 그대로 사용).

## 단계 1 — 사전 점검 (반드시 인터뷰보다 먼저)

1. **`D:\AX\hub\RESOURCES.md`를 읽는다.** 새 프로젝트가 외부 데이터를 쓸 가능성이 있다면, 이미 등록된 자원과 중복 여지가 있는지 본다.
   - 한국 주식 데이터 → 키움 REST 또는 네이버 finance 이미 있음
   - 코인 캔들 → `crypto/fetch_klines.py` 이미 있음
   - 일본/미국 주식 → Minkabu / Yahoo 이미 있음
   - 텔레그램 알림 → `kiwoom/notify.py` 라우터 4채널 이미 있음
2. **`D:\AX\hub\INDEX.md`를 읽어** 도메인 이름 충돌 여부와 현재 운영 중인 도메인 파악.
3. 중복 자원이 발견되면 사용자에게 인터뷰 전에 한 번 알린다. "○○ 도메인의 △△가 이미 있는데, 재사용할까요 아니면 별도로 구축할까요?"

## 단계 2 — 인터뷰 (4문항, AskUserQuestion으로 한 번에)

사용자 발화에 이미 답변이 포함돼 있으면 인터뷰 생략 가능. 그렇지 않으면 다음 4가지를 한 번에 물어본다.

1. **목표** — 이 프로젝트로 뭘 달성하고 싶은지 (1~2문장)
2. **데이터/외부 자원** — 어떤 외부 API나 데이터 소스가 필요한지 (RESOURCES.md 자원 재사용? 신규 도입?)
3. **Phase 1 첫 작업** — 가장 먼저 만들 모듈 또는 검증할 것 1개
4. **(선택) 기존 보유 자산** — 이미 수집해둔 데이터/코드/문서가 있는지

## 단계 3 — 산출물 생성

### 3-1. 디렉토리 + PROJECT_PLAN.md

### 3-2. hub/INDEX.md 자동 추가

- 트레이딩 도메인인지 비트레이딩인지 인터뷰 답변에서 판단
- 해당 표에 새 행 추가:

```
| **<프로젝트명>** | 시작 단계 | <Phase 1 첫 작업> | (미정) | [PROJECT_PLAN](../<프로젝트명>/PROJECT_PLAN.md) |
```

- INDEX.md 하단 "통합 작업 진행" 섹션에 한 줄 추가:

```
- <YYYY-MM-DD>: 새 도메인 `<프로젝트명>` 등록 (PROJECT_PLAN.md)
```

### 3-3. hub/RESOURCES.md 갱신 (조건부)

- 새 외부 자원을 **신규 도입**하는 경우에만 카탈로그에 추가
- 기존 자원 재사용은 **추가하지 않음** (이미 있음, 중복 방지)
- 새 자원의 항목: 도메인/URL/사용 위치(미정 가능)/인증/제한/상태(검토 중 → 운영)

## 출력 순서

채팅 응답에 다음 순서대로 표시:

1. **사전 점검 결과** — 중복 자원 발견 시 보고 (없으면 1줄 "중복 없음")
2. **인터뷰 결과 요약** — 4문항 답변을 2~3줄로 정리
3. **생성 항목** — PROJECT_PLAN.md 경로, INDEX.md 갱신, (필요 시) RESOURCES.md 갱신
4. **다음 단계 안내** — "이제 D:\AX\<프로젝트명>에서 인터랙티브로 시작하세요" + Phase 1 첫 작업 다시 한 번

## 하지 말 것

- **빈 .py 파일 자동 생성 금지** — 구조는 인터랙티브로 결정
- **다른 도메인 코드 자동 복제 금지** — kiwoom 모듈 구조가 유사하다고 us에 복붙하지 않기. 사용자가 명시적으로 요청한 경우에만
- **외부 API 호출 코드 자동 작성 금지** — 인증 정보 필요, 사용자 확인 후
- **MEMORY.md에 자동 등록 금지** — 사용자가 프로젝트 본격 진행 후 wrap-up skill 단계에서
- **PROJECT_STATUS.md를 처음부터 만들지 말 것** — PLAN으로 시작, 운영 단계 진입 시 사용자 결정으로 STATUS로 전환

## 운영 단계 진입 (별도 흐름)

PROJECT_PLAN.md가 Phase 1을 완료하고 운영 단계로 들어가면:

- 사용자가 "STATUS 파일로 전환해줘" 또는 wrap-up skill에서 자동 갱신
- PROJECT_PLAN.md → PROJECT_STATUS.md로 리네임 또는 별도 생성 (사용자 선택)
- INDEX.md의 링크 자동 갱신

이 흐름은 start-project skill의 책임 아님 (wrap-up 또는 수동).
