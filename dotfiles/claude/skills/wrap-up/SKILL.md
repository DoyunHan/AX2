# Wrap-up

Context를 clear하기 전에 현재 세션 작업을 다음 세션이 바로 이어받을 수 있는 상태로 정리한다.

## 작동 원칙

- 사용자는 D:\AX\ 아래 다중 프로젝트(kiwoom, wti, crypto, japan, us, quant, medical, poker, english, unity 등)를 동시 운영한다. **이번 세션에서 실제로 손댄 프로젝트에만** 적용한다. 전 프로젝트를 훑지 않는다.
- 작업 흔적은 대화 컨텍스트와 git 변경분에서 파악한다. 외부 도구로 따로 조회하지 않는다.
- 출력은 짧게. 사용자가 컨텍스트를 잃기 직전이므로 결과 요약은 핸드오프 블록 하나로 충분하다.

## 5단계 루틴

각 단계의 변경 사항을 사용자에게 보고하기 전에 **먼저 변경할 항목 후보를 한 줄씩 나열**하고, 사용자 확인 후 적용한다. 정리 자체가 잘못 기록되면 손해가 크므로 무단 진행 금지.

### 1) PROJECT_STATUS.md 갱신

- 이번 세션에서 진척이 있었던 프로젝트의 `PROJECT_STATUS.md`(또는 그에 준하는 파일 — 예: `DAYTRADE_STATUS.md`, `<프로젝트>/STATUS.md`)를 찾는다.
- 파일이 없으면 만들지 말고 사용자에게 "이 프로젝트는 STATUS 파일이 없는데 만들까요?"라고 물어본다.
- 갱신 내용: 오늘 한 일 / 현재 상태 / 다음 단계 / 막힌 점(있을 때만). 마크다운 헤더 형식은 기존 파일 스타일을 따른다.
- 날짜는 `currentDate`를 절대 날짜로 기입 (예: 2026-05-15).

### 2) 다음 세션 핸드오프 요약

새 대화 시작 시 사용자가 그대로 붙여넣어도 작업이 이어지는 짧은 블록을 출력한다. 형식:

```
## 핸드오프 (YYYY-MM-DD)

**작업 프로젝트:** <경로>
**이번 세션 결과:** <2~4줄>
**다음 시작 지점:** <첫 줄에 무엇을 해야 하는지 명령형으로>
**참고:** <필요한 파일/메모리/링크 — 없으면 생략>
```

- 핸드오프는 **사용자에게 채팅으로 출력**한다. 파일로 따로 저장하지 않는다 (사용자가 명시적으로 요청한 경우에만).
- 본문 4줄 이내. "이번 세션 결과"에 코드 변경/백테스트 수치/결정 사항을 모두 압축.

### 3) 커밋 안 된 변경사항 점검

- `git status` 와 `git diff --stat`을 PowerShell로 실행한다.
- 결과를 분석해서 **현재 세션에서 만든 변경분**과 **세션 시작 시점부터 이미 있던 변경분(working tree에 떠있던 것)**을 구분해 보고한다.
- 커밋 여부는 사용자가 결정. 사용자가 "커밋해줘"라고 명시하지 않는 한 직접 commit/add 하지 않는다.
- **커밋할 때는 반드시 명시 pathspec** — `git commit -- <이번 세션 파일들>`. `git add -A`/`git commit -am` 금지. D:\AX는 ~20개 세션이 같은 워킹트리를 공유해 광역 add는 다른 세션의 미커밋 변경을 한 커밋에 포섭한다(history 오귀속). (`feedback_multisession_shared_file_safety`)
- secrets 의심 파일(.env, *.key, credentials*, *_key.json 등)이 staged/modified에 보이면 별도 경고.
- **dotfiles 드리프트 점검** — 이번 세션에서 `~/.claude/`의 스킬·`settings.json`·`statusline.sh`를 수정했다면, `D:\AX\dotfiles\claude\`(+미러 `D:\AX2\dotfiles\claude\`)의 동명 파일과 diff해 **전파 안 된 변경을 경고**한다. 수동 동기화라 빠지기 쉬움(실제 누락 이력 있음). 단 자동 복사는 하지 말 것 — `settings.json`은 다중 세션이 bare로 재기록해 **키 순서만 바뀌는 노이즈**가 흔하니, 내용이 실제로 바뀐 파일만 전파 대상으로 보고한다. ([[reference_dotfiles_claude]])

### 4) 메모리 업데이트

- 이번 세션에서 새로 알게 된 / 바뀐 / 폐기된 사실 중 **다음 세션에 필요한 것만** 골라 메모리에 반영한다. 사소한 진척은 STATUS 파일에 들어갔으니 메모리에 또 쓰지 않는다.
- 후보 분류:
  - **새 메모리 필요**: 이번 세션에서 처음 학습한 user/feedback/project/reference 사실
  - **기존 메모리 수정**: MEMORY.md 또는 개별 파일의 description/본문이 실제 상태와 어긋남 (예: "보류" → "재개", 수치 갱신)
  - **기존 메모리 폐기**: 더 이상 유효하지 않거나 더 정확한 메모리로 대체됨
- 적용 전 사용자에게 후보 리스트를 보여주고 확인받는다. "이 4건을 [추가/수정/삭제]할게요" 형태.
- MEMORY.md는 200줄 후 잘리므로 한 줄짜리 index만 유지. 본문은 별도 파일에.
- **MEMORY.md는 전체파일 Write 금지 — 항상 `Edit`로 한 줄만 추가/수정**, 그리고 이 4단계를 **메모리 작업의 맨 마지막에** 둔다. "file modified since read"가 뜨면 다른 세션이 동시 편집한 것이니 재read 후 재시도(덮어쓰지 말 것). (`feedback_multisession_shared_file_safety`)

### 5) 다음 세션 재개 프롬프트 생성

핸드오프 블록 바로 뒤에, 사용자가 `/clear` 후 그대로 붙여넣어 작업을 이어갈 수 있는 **한 줄 명령**을 출력한다.

형식:

```
---

### 다음 세션 재개 (붙여넣기용)
1) /clear
2) 아래를 그대로 붙여넣으세요:

> <프로젝트 경로> 작업 이어서. <진입 파일/섹션 또는 첫 액션>.
> (참고 메모리: <slug1>, <slug2>)
```

구성 규칙:

- **첫 줄**: `<프로젝트 경로> 작업 이어서. <진입점>.` — 70자 이내 권장. 명령형. 진입점은 (a) STATUS 파일의 특정 섹션, (b) 다음 액션을 한 문장으로 적은 것 중 더 구체적인 쪽.
- **둘째 줄(선택)**: `(참고 메모리: ...)` — 새 세션이 우선 읽어야 할 메모리 슬러그. MEMORY.md에 이미 등록된 슬러그만 사용. 보통 1~3개.
- STATUS 파일이 없는 경우(생성 거부 포함): 진입점은 "이번 세션 결과의 마지막 액션 다음에 해야 할 일"을 한 문장으로 작성.
- `/clear`를 Claude가 직접 실행하려고 시도하지 않는다. 안내문만 출력한다.

## 출력 순서

채팅 응답에 다음 순서대로 표시:

1. **점검 결과 요약** — 어떤 프로젝트가 영향받았는지 1줄
2. **STATUS 갱신 제안** — 어느 파일에 어떤 섹션을 쓸지 (사용자 확인)
3. **git 변경 요약** — 파일 N개 변경, 그중 secrets 의심 여부
4. **메모리 후보** — 추가/수정/삭제 리스트 (사용자 확인)
5. (확인 후 적용)
6. **핸드오프 블록** — 위 형식대로
7. **재개 프롬프트 + /clear 안내** — 위 5)단계 형식

## 하지 말 것

- 사용자가 손대지 않은 프로젝트의 STATUS/메모리를 "정리해두는 김에" 함께 수정하지 말 것
- 새 문서(요약 .md 파일 등)를 working dir에 만들지 말 것 — 사용자가 명시적으로 요청한 경우에만
- 추측으로 메모리를 갱신하지 말 것 — 컨텍스트에 명백한 근거가 있는 사실만
- git commit/push/branch 작업을 자동으로 하지 말 것
- `/clear`를 자동으로 호출하려 시도하지 말 것 — CLI 빌트인이라 Claude가 호출 불가. 재개 프롬프트와 안내문만 출력.
